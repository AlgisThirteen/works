RENDER CLOUD DEPLOYMENT

This project now includes render.yaml for a small Render web service deployment.

Why Render
- Render supports Docker-based web services defined in render.yaml.
- Blueprint env vars can use sync: false so secrets are entered in the Render dashboard instead of being committed to source control.
- Render sets a runtime PORT environment variable for web services, with default port 10000 if not otherwise overridden.

Files added for Render
- render.yaml
- start.sh
- Dockerfile updated to use start.sh

What render.yaml does
- Creates one web service named kalshi-app
- Uses runtime: docker
- Uses plan: starter
- Uses region: oregon
- Sets health check path to /health
- Prompts you to provide KALSHI_API_KEY_ID and KALSHI_PRIVATE_KEY at creation time via sync: false

Important secret choice
- For Render, prefer KALSHI_PRIVATE_KEY as an environment variable containing the full PEM text.
- The app now supports either:
  1) KALSHI_PRIVATE_KEY as inline PEM text
  2) KALSHI_PRIVATE_KEY_PATH as a filesystem path
- In Render, the inline env var is the easiest option.

How to deploy on Render
1. Push this project to GitHub.
2. In Render, create a new Blueprint or deploy from the repository.
3. Render detects render.yaml.
4. During first setup, provide values for:
   - KALSHI_API_KEY_ID
   - KALSHI_PRIVATE_KEY
5. Leave KALSHI_ENV as demo for safe testing, or change to production later.
6. Deploy.

After deploy
- Visit /health first.
- Confirm:
  - auth_configured is true
  - private_key_exists is true
  - private_key_source is env
- Then test /balance and /positions.

Notes
- Do not paste secrets into render.yaml itself.
- If you later rotate keys, update them in the Render dashboard Environment page.
- If you want to use Render secret files instead, you can upload a secret file and set KALSHI_PRIVATE_KEY_PATH accordingly, but the inline env var path is simpler for this app.
